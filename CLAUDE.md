# 项目纪律

## 改代码必须加测试

任何对生产代码的修改（新功能、bug 修复、重构）都必须在同一次提交里附带测试。
没有测试的改动不算完成，不要提交，也不要声称任务完成。

- 修 bug：先写一个能复现该 bug 的测试，确认它失败，再写修复，确认它通过。
- 加功能：测试覆盖至少一条主路径 + 一个边界情况。
- 重构：依赖现有测试保护行为；如果现有测试不足以覆盖被改动的行为，先补齐再重构。

唯一例外：纯文档改动（README、注释）、纯格式化、纯依赖升级。其它情况一律加测试。

## 测试不准启动有头浏览器

如果需要驱动真实浏览器，**必须以 headless 模式运行**（Playwright / Puppeteer / 其它），
绝不允许在测试或自动化流程中弹出有头窗口（即不要 `headless: false`、不要 `--headed`、
不要依赖本机图形环境）。

除此之外，**怎么测试由 agent 自行判断**——单元测试、jsdom、headless E2E、契约测试、
快照测试、纯函数 fixture 测试等都可以，根据被改动代码的性质选最合适的方式。

理由：本项目大部分时候在无图形环境（CI、容器、远程会话）里运行，启动有头浏览器会直接
失败或卡住。
